Grocery Store Inventory & Sales Management

SDG Alignment

  • SDG 2 – Zero Hunger
  • SDG 12 – Responsible Consumption & Production
  • SDG 8 – Decent Work & Economic Growth

Problem

  • Difficulty tracking product inventory efficiently
  • Products are not arranged properly, making shelf organization inefficient.
  • Products expiring before being sold
  • Shelves do not highlight soon-to-expire products, causing losses.
  • Overstocking or stockouts due to poor planning
  • Fast-selling products are not identified early, causing stockouts.
  • Long checkout queues affecting customer experience
  • Need for dynamic product sorting (price, rating, expiry)

System Requirements

  • Enable real-time tracking of all product inventory levels.
  • Support efficient arrangement of items for better shelf organization.
  • Automatically detect and highlight products nearing expiry.
  • Organize shelf placement based on expiry priority to reduce waste.
  • Continuously monitor stock levels to avoid overstocking or stockouts.
  • Identify and display top-selling items for timely replenishment.
  • Ensure smooth and fair checkout flow using an orderly queue process.
  • Allow dynamic sorting of products by price, rating, or expiry.

How it is implemented

Feature Algorithm / Data Structure implementatioin steps efficiency
Real-time inventory tracking Hash Table • Store each product as a key–value pair (SKU → details).
• Update stock instantly on every sale/arrival.
• Retrieve product info in O(1) time for live inventor		 Hash table: O(1) average
Shelf arrangement optimization QuickSort • Extract all products from inventory.
• Sort them by category/size using QuickSort.
• Arrange shelves based on the sorted list.		 QuickSort:O(n log n) average
Identify soon-to-expire products Min-Heap • Insert products into a Min-Heap using expiry as priority.
• Always read the root to get the nearest-expiring item.
• Trigger alerts when expiry is close.
 Min-Heap:O(log n) average
Expiry-based shelf arrangement QuickSort • Sort products by expiry date using QuickSort.
• Place soon-to-expire items at the front.
• Re-sort when new stock arrives.
 QuickSort:O(n log n) average
Track understock/overstock Hash Table • Continuously monitor stock levels with instant access to any item.
• Compare quantity with min/max thresholds.
• Auto-alert for reorder or overstock adjustments.
 Hash Table:O(1) average
Identify best-selling products Max-Heap • Insert products with sales count as heap priority.
• Update heap after each sale.
• Root gives the top-selling item for faster restocking.
 Max-Heap:O(log n) average
Efficient checkout queue handling FIFO Queue • Add customers to the queue on arrival.
• Serve using FCFS by dequeueing.
• Open extra counters when queue grows.
 Queue:O(1) average
Dynamic product sorting MergeSort / HeapSort / QuickSort • Sort by price using MergeSort for stability.
• Sort by rating using HeapSort for fast selection.
• Sort by expiry using QuickSort for efficiency
 MergeSort / HeapSort / QuickSort:O(n log n) average

Healthcare Appointment Scheduling & Emergency Dispatch

SDG Alignment:

SDG 3 – Good Health & Well-being
SDG 11 – Sustainable Cities & Communities

Probelms:

  • Patients face long waiting times due to poor priority handling.
  • Searching patient records is slow because data is not efficiently organized.
  • Doctors are sometimes double-booked due to lack of conflict detection.
  • Appointment rescheduling becomes confusing without proper queue order.
  • Emergency cases are delayed because they are not handled with immediate priority.
  • Appointment tokens are disorganized, causing timing and sequencing issues.
  • Finding symptoms in large patient notes is slow and inefficient.
  • Insurance verification takes time because records are not balanced or sorted.
  • Hospital analytics (patients, bills, tests) is slow due to inefficient range queries.
  • Patients and staff struggle to find the shortest route inside the hospital.

    • System Requirements

    • Automatically prioritize patients based on severity and arrival time.
    • Store patient records in a structure that supports instant searching.
    • Detect and prevent doctor scheduling conflicts before confirming appointments.
    • Maintain a clear, ordered queue to allow smooth appointment rescheduling.
    • Place emergency cases at the highest priority for immediate handling.
    • Organize appointment tokens in correct chronological order.
    • Support fast keyword and pattern-based searching inside patient notes.
    • Keep insurance records balanced and sorted for quick verification.
    • Enable fast analytics using optimized range-query processing.
    • Guide users with accurate shortest-path navigation through the hospital.

    How it is implemented

    Feature Algorithm / Data Structure Implementation Steps efficiency
    Long patient waiting times Skip List • Assign each patient a priority (emergency/high or normal/low).
    • Insert the patient into the Skip List based on priority.
    • Use multiple “levels” to quickly locate correct position.
    • Doctor always picks the first (highest priority) patient.     		skiplist:O(log n)average
    Slow patient record searching Binary Search Tree (BST) • Use patient ID or name as the key.
    • Insert new records in sorted order in the BST.
    • Search by moving left/right based on key comparison.
    • Access records without scanning all entries.     		BST:O(log n)average
    Doctor allocation conflicts Union-Find (Disjoint Set Union) • Treat each doctor as a node.
    • Group doctors under departments using union().
    • Before assigning, run find() to check assignments.
    • If both assignments belong to same set → conflict detected.     		Union-Find : o(1)average
    Appointment rescheduling confusion Queue • Add appointments to the back of the queue.
    • Serve patients from the front in order.
    • On rescheduling, reinsert patient in queue properly.
    • Ensures first-come-first-served flow.     		Queue:O(1) average
    Emergency patient handling delays Stack • Push new emergency cases to the top of the stack.
    • Doctor treats latest emergency first (pop).
    • Remove emergency entry after treatment.
    • Ensures instant attention for emergencies.     		stack:O(1) average
    Unsorted appointment tokens Selection Sort • Collect all tokens.
    • Repeatedly find the earliest token.
    • Place it at the beginning.
    • Continue until list is fully sorted.     		selection sort:O(n²)average
    Difficulty searching symptoms in notes KMP Algorithm • Doctor enters keyword.
    • Build LPS (Longest Prefix Suffix) table.
    • Scan the notes using KMP pattern matching.
    • Locate symptoms instantly without re-checking characters.     		KMP algorithm:O(n) average
    Slow insurance verification Red-Black Tree • Insert each policy into the Red-Black Tree.
    • Tree self-balances on insert/delete.
    • Search for policy number or expiry quickly.
    • Provides consistent, fast lookup.     		Red-Black Tree:O(log n)
    Slow medical analytics (counts, bills, tests) Fenwick Tree (Binary Indexed Tree) • Store daily statistics in BIT.
    • Update counts using BIT update operation.
    • Query using prefix(end) − prefix(start).
    • Get instant totals for any time range.     		Fenwick Tree:O(log n)average
    Difficulty finding shortest route in hospital Dijkstra’s Algorithm • Represent floors, rooms, halls as graph nodes.
    • Connect nodes with weighted edges (distance/time).
    • Run Dijkstra from source when destination selected.
    • Display shortest/fastest route.     		Dijkstra’s Algorithm:O(n^2)

    Restaurant Order & Delivery System

    SDG Alignment:
    SDG 8 – Decent Work & Economic Growth
    SDG 11 – Sustainable Cities & Communities
    SDG 12 – Responsible Consumption & Production

    Problem:

    SDG Alignment:
    SDG 8 – Decent Work & Economic Growth
    SDG 11 – Sustainable Cities & Communities
    SDG 12 – Responsible Consumption & Production

    Problem:

    Restaurants face challenges such as:

    • The system struggles to store and organize a large number of orders efficiently.
    • Retrieving customer preferences and past order details is slow and inefficient.
    • Accessing menu details like price, category, and ingredients takes longer as the menu grows.
    • Slow searching of dishes based on keywords or ingredients in descriptions.
    • Inability to quickly determine which dishes are trending or most ordered.
    • Sorting dishes by rating or popularity for recommendations becomes slow.
    • The system struggles to compute the fastest delivery route under varying traffic conditions.
    • Calculating distances between multiple delivery locations repeatedly wastes time.

    system-requirements:

    • Store and manage orders using a fast, balanced, time-sorted structure.
    • Provide quick retrieval of customer names, preferences, and past orders.
    • Allow constant-time access to menu items and their details.
    • Support fast keyword or ingredient search within dish descriptions.
    • Identify and update popular dishes instantly as orders come in.
    • Efficiently sort dishes based on ratings or popularity for recommendations.
    • Compute the shortest and fastest delivery route even under traffic variations.
    • Maintain all-pairs shortest paths for repeated delivery operations.

    How it is implemented:

    Feature Algorithm / Data Structure Implementation Steps efficiency
    Order Storage & Management AVL Tree • Store each order as a node in an AVL Tree.
    • Insert new orders while maintaining tree balance.
    • Perform left/right rotations when height difference exceeds 1.
    • Retrieve orders quickly using in-order traversal.
    • Keeps orders sorted by timestamp or order ID.     		AVL Tree:O(log N)average
    Customer Preferences & Order History Trie • Store customer names and frequent dishes as character nodes.
    • Insert names/dishes letter-by-letter.
    • Retrieve customer history using prefix search.
    • Enables auto-suggestions and fast lookups.     		Trie:O(L)average
    Menu Lookup Lookup Table • Store menu items using indexed keys.
    • Direct O(1) access to price, ingredients, and details.
    • Update entries instantly when menu changes.     		Lookup Table:O(1)average
    Keyword Search in Menu Boyer–Moore Algorithm • Preprocess keyword to create bad-character and good-suffix tables.
    • Scan menu descriptions using Boyer–Moore pattern matching.
    • Skip unnecessary comparisons for faster searching.
    • Return dishes whose descriptions contain the keyword.     		>Boyer–Moore Algorithm:O(N / M) (average)N = text length, M = pattern length
    Trending or Most Ordered Dishes Max-Heap • Maintain frequency count for each dish.
    • Insert/update dish in Max-Heap based on popularity.
    • Root always shows the most popular dish.
    • Extract max to display trending dishes.     		Max-Heap:O(log N)average
    Dish Recommendation Sorting Insertion Sort • Store dishes with ratings or popularity scores.
    • Compare each dish with previously sorted items.
    • Insert each dish into its correct sorted position.
    • Generates Top-N sorted recommendations.     		Insertion Sort:O(N²)average
    Best Delivery Route Optimization Bellman–Ford Algorithm • Model delivery areas as a weighted graph.
    • Initialize distance to all nodes from the source.
    • Relax all edges repeatedly to find shortest paths.
    • Supports negative weights (traffic delays).
    • Returns best/fastest delivery route.     		Bellman–Ford Algorithm:O(V × E)average
    Precomputed Distance Between Locations Floyd–Warshall Algorithm • Represent all delivery points as graph nodes.
    • Create a distance matrix for every node pair.
    • Apply: dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]).
    • Precomputes all-pairs shortest path.
    • Ideal for repeated deliveries & route planning.     		Floyd–Warshall Algorithm:O(N³)average

    Smart Waste Collection Optimization System

    SDG Alignment

    • SDG 11 – Sustainable Cities & Communities
    • SDG 12 – Responsible Consumption & Production
    • SDG 3 – Good Health & Well-Being

    Problem

    Problems Faced by Cities

    • Overflowing bins due to irregular waste collection
    • Inefficient waste collection routes causing long travel times
    • Sudden waste spikes during festivals or seasonal events
    • Staff inability to manually check all bins across the city
    • Overflow incidents not being detected on time
    • Improper allocation of trucks and staff based on waste load
    • Difficulty identifying the exact locations of bins and truck positions

    System Requirements

    • Requires real-time monitoring of bin fill levels
    • Needs optimized routes for waste collection trucks
    • Needs forecasting support from historical data
    • Requires a centralized dashboard showing all bins
    • Requires automatic alerts when bins exceed thresholds
    • System must analyze patterns to allocate resources
    • Needs GPS-based mapping & location lookup
    Feature Algorithm / Data Structure Implementation Steps efficiency
    Overflowing bins due to irregular waste collection Binary Tree • Store bin fill levels in sorted order.
    • Quickly find bins nearing overflow.
    • Update values as sensors send new data.     		Binary Tree:O(log n)average
    2–3 Tree • Insert continuous sensor readings.
    • Balanced structure ensures fast overflow detection.
    • Handles large real-time data smoothly.     		2–3 Tree:O(log n)average
    Bubble Sort • Sort bins by fill level.
    • Move nearly overflowing bins to the top.
    • Re-run after every sensor update.     		Bubble Sort:O(n²)average
    Inefficient waste collection routes DFS • Explore all road connections.
    • Understand complete path structure.
    • Map unreachable or isolated areas.     		DFS:O(V + E)
    BFS • Find shortest hop-based route from truck → bin.
    • Optimize travel time in dense zones.
    • Recalculate quickly during route changes.     		DFS:O(V + E)
    Kruskal's Algorithm • Build a Minimum Spanning Tree of bins.
    • Group nearest bins to reduce distance.
    • Minimize total route cost.     		Kruskal's Algorithm:O(E log E)
    Sudden waste spikes during festivals Binary Tree • Store historical fill-level data by date/time.
    • Compare patterns with current data.
    • Predict when spikes may occur.     		Binary Tree:O(log n)average
    2–3 Tree • Handle large historical datasets efficiently.
    • Maintain balanced access for trend analysis.
    • Support fast pattern detection.     		2–3 Tree:O(log n)average
    Staff cannot monitor all bins manually DFS • Scan all bins zone by zone.
    • Refresh statuses across the entire network.
    • Detect unresponsive or faulty bins.     		DFS:O(V + E)
    BFS • Find nearest bins quickly for dashboards.
    • Update live status in real time.
    • Assist in quick decision-making.     		DFS:O(V + E)
    Binary Tree • Store bin metadata (ID, location, last update).
    • Retrieve any bin details instantly.
    • Improve dashboard responsiveness.     		Binary Tree:O(log n)average
    Overflow incidents are not detected quickly Bubble Sort • Sort bins by fill level from high → low.
    • Identify dangerous bins immediately.
    • Reduce reaction time for staff.     		Bubble Sort:O(n²)average
    2–3 Tree • Store real-time sensor values in balanced form.
    • Ensure stable threshold checks.
    • Handle huge data streams efficiently.     		2–3 Tree:O(log n)average
    Binary Tree • Store alert rules and thresholds.
    • Compare values quickly.
    • Trigger instant notifications.     		Binary Tree:O(log n)average
    Improper truck and staff allocation Binary Tree • Store zone-wise waste levels.
    • Compare which zones need more trucks.
    • Allocate resources based on load.     		Binary Tree:O(log n)average
    2–3 Tree • Maintain long-term waste patterns.
    • Predict future staff requirements.
    • Plan optimal truck distribution.     		2–3 Tree:O(log n)average
    Difficulty identifying bin/truck locations Binary Tree • Store GPS coordinates of each bin/truck.
    • Allow fast lookup by ID.
    • Improve location tracking accuracy.     		Binary Tree:O(log n)average
    2–3 Tree • Manage continuous GPS updates.
    • Keep searches fast even as data scales.
    • Support real-time positioning.     		2–3 Tree:O(log n)average
    BFS • Find nearest bins to a truck's location.
    • Suggest shortest movement path.
    • Improve navigation efficiency.     		DFS:O(V + E)
